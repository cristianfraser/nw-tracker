# Agent notes (nw-tracker)

## Language and i18n

- **Code in English**: identifiers, comments, commit messages, and TypeScript/React source should be written in English.
- **UI copy via i18n (es + en)**: the UI is bilingual. All user-facing strings live in **`client/src/i18n/master.json`** — the single source of truth, same nesting as the locale files but every leaf holds both languages side by side: `{ "en": "Save", "es": "Guardar" }`. `locales/es.json` / `locales/en.json` are **generated** from it by `npm run i18n:generate -w nw-tracker-client` (`client/scripts/i18n-generate.mjs`); never hand-edit the locale files — `client/src/i18n/localeSync.test.ts` fails on any drift. The master stays nested (not flat dotted keys) because some leaf keys contain literal dots (e.g. `panelAccounts.addAccount."accountType.equity"`). Do not hardcode Spanish or English labels in components except via `t('…')` / `i18n.t('…')`.
- **Translation keys in English**: use dotted paths such as `dashboard.cards.netWorth`, `brokerage.groups.mutual_funds`. Code identifiers use English slugs (`mutual_funds`, `real_estate`); display copy lives in `master.json`.
- **Shared labels**: use helpers in `client/src/i18n/index.ts` (e.g. `brokerageGroupLabel`) so the sidebar, dashboard cards, and charts read the same keys.
- When adding UI text, add the key (both languages) to `master.json` first, run `npm run i18n:generate -w nw-tracker-client`, then reference it from code. Migrate existing hardcoded strings when touching an area.
- **UI language preference**: `client/src/languagePreference.ts` (localStorage `nw-tracker.language`), seeded once from `navigator.language` (Spanish → `es`, anything else → `en`; non-browser/vitest contexts stay `es` so label assertions are deterministic), then the stored value is authoritative. Changed from `/panel/settings` via `DisplayPreferencesContext.setLanguage` (calls `i18n.changeLanguage`, persists, and keeps `<html lang>` in sync; the context change re-renders `AppTree` top-down). The decimal separator remains an independent setting. **Do not cache `t()` results** in `useMemo`/state without the language in the deps — translate at render time.
- **Dates**: numeric dates render as ISO `YYYY-MM-DD` (timestamps `YYYY-MM-DD HH:MM`) in every language — never `dd-mm-yy` / `toLocale*` short styles, which are ambiguous across locales. Month-name labels (`dic 2026` ↔ `Dec 2026`, chart ticks `dic 16` ↔ `Dec 16`) go through `client/src/formatDateLabel.ts`, which reads `i18n.language` at call time — same no-caching rule as `t()`.
- **Enforced by `npm run check:conventions`** (root; also runs at the end of root `npm run typecheck`): flags accented Spanish in client string literals/JSX (comments exempt), `Intl.NumberFormat` outside `format.ts`, and hardcoded-locale `toLocaleString` without date options. Escape hatch: `// convention-ok: <reason>` on the line.
- **Number formatting**: all numbers (CLP, USD, UF, units, percents) share one decimal-separator convention from `client/src/numberFormatPreference.ts` (localStorage `nw-tracker.decimalSeparator`, seeded from the IANA timezone; toolbar toggle overrides). Always format via `client/src/format.ts` helpers — never `Intl.NumberFormat` / `toLocaleString` with a hardcoded locale for numbers (dates: see the Dates bullet above — ISO numerics + `formatDateLabel.ts` month names). A separator change re-renders (not remounts) the tree via `AppTree`; **do not cache formatted strings** in `useMemo`/state unless `decimalSeparator` is in the deps — memoize raw numbers and format at render time.

## Fail fast — no runtime fallbacks

- **Fix data, don’t paper over it**: missing links, mismatched totals, ambiguous parse fields, and inconsistent imports are data or pipeline problems. Repair imports, migrations, or source files — do not add runtime fallbacks, `try/catch` swallowing, or “best guess” edge-case branches to keep the UI green.
- **Throw on mismatch**: when parsing, importing, or reconciling two sources (e.g. statement vs ledger, cert vs units, card breakdown vs NAV), invalid or inconsistent input should **throw** (or return a hard error to the client). Do not return `null`, `0`, empty arrays, or alternate code paths as silent defaults.
- **Avoid fallbacks**: no filename/statement-date guesses when a canonical field is required; no alternate formulas when the primary calc fails; no `catch { return fallback }`. Existing docs (e.g. CC matrix month = `period_to` only) follow this rule — extend it to new code.
- **Legitimate exceptions**: user-facing validation messages, optional UI features that are truly optional by product design, and infrastructure retries (network, file locks) — not business-logic substitutes for bad DB state.
- **Notes are provenance (2026-07):** movement/account `note` text is for humans; machine-readable state lives in columns/tables, written at import/create time. Runtime never branches on note substrings and never reads `cfraser/` files on request paths. When a runtime guard exists only to paper over a legacy data state: verify the DB state once, fix rows if needed, then delete the guard (fail fast if the state reappears). Structured homes: `movements.flow_kind` (deposit/mortgage/brokerage kinds), `depto_payments`, `movement_mirror_merges`, `accounts.import_key`, `credit_card_account_config.card_last4` / `.nav_retired` / `.superseded_target_last4` (the config `notes` column was dropped in migration 160).

## CC chart / Pasivos convention

**"owed on that date", not billing-month totals.** Per-card chart lines, the Pasivos group Total, `accountMarkClpAtYmd`, and the net-worth dashboard Pasivos line all read stored `valuations` for historical dates (live billing balance for today) via `latestCreditCardValuationRowAsOf`; there is no separate "Saldo pasivos" ref line (removed — Total carries the same series). The month-end points `upsertCreditCardValuationsFromLedger` writes are billing-detail `balance_total` = owed-at-cierre (the `− cuota a pagar next mes` term nets out the cuotas billed at that close, statement-exact) + post-cierre window activity (charges +, PAGOs −) — evidence-based, re-synced after web-paste repairs in `mergeCcAccountFromParsedRows`. **Today-stamp caveat (2026-07-22):** `upsertCreditCardValuationsFromLedger` also stamps a `today` valuation on every CC write; month-end rows re-sync on later runs but past today-stamps never do — they freeze whatever the live formula said that day and become permanent daily-owed anchors. After fixing a CC balance convention, audit recent stamps against the corrected line walk and delete provably-wrong residue (six stale ·7817 rows from the missing-PAGO era were purged 2026-07-22).

- **One schedule framing (2026-07-13): the installment plan schedule is facturación/statement months everywhere.** `purchaseFirstDueYm` + index gives the month each cuota **bills**; per-month plan sums equal the statements' billed-cuota sums exactly, and `cuotaAPagarNextMesClp` / facturaciones read the billing month itself (never +1). Anchor precedence: cuota-01 statement month → cuota-00 preamble's statement month + 1 → stored `first_due_month` (web-paste nudge, write-once evidence) → manual guess = the purchase's **own** cycle (date-based so it can't drift; the next statement's evidence relocates it either way — cuota-01 pins the month, a cuota-00 preamble moves it to statement + 1). Nota-cancelled plans (full-principal NOTA DE CREDITO matched by posting **statement month** — banks backdate the transaction date, never the statement — or by a strictly-later transaction date) are excluded from every schedule aggregate via the loader's `schedulePurchases`; live cupo therefore matches the printed «cupo utilizado en cuotas». A fully **settled** ledger no longer force-actives a card (`creditCardBillingDetailInactive` requires outstanding principal, else the tail heuristic decides).
- **Cuotas por facturación (`payload.months`)**: one row per facturación event — `month` (facturación) + `pay_by_date` (statement PAGAR HASTA when closed; derived ~10th of the next month for open/projected) + `total_clp` + `debt_after_clp` (suffix sum; last row 0). Rows roll off once their pay-by passes (Chile day — the same clock that flips a fully-paid purchase from active to completed); `totals.next_calendar_month` reports the **pay** month of the next payment event. Purchase tables show `purchase_date` + `purchase_billing_month` (billing config applied to the purchase date), not the calendar purchase month.
- **Months after the open facturación are plan-only projections** (`projected: true` rows from `appendProjectedBillingDetailRows`), **pay-frame**: facturado is null (nothing billed yet) and saldo/cupo at month-end M = plan remainder after (M−1)'s close — a cycle's cuotas leave the debt on their **pay-by** (~10th of M+1), not at the close that bills them. One facturación step per month, flat from today through the open month-end, plus a trailing zero month after the final pay-by. Identity: chart month-end M = calendar-table row M's cuota + `debt_after_clp`. Closed months keep the statement frame (printed cupo; billed-unpaid cuotas ride inside facturado) and today keeps the live value — the seam is smooth because bank-frame(M) ≡ pay-frame(M+1). `latestBillingDetailRow` skips projected rows so the live «Balance total» / today's Pasivos point stays the open-month rolled balance.

## Global sync scheduler

- In-process scheduler (`server/src/globalSyncScheduler.ts`): polls every 15 minutes **only while any source is stale**; when all are fresh, stops polling and wakes at the earliest source `next_sync` wall time.
- **Only the user's primary app instance runs background jobs.** Any secondary dev server sharing the real DB — every `.claude/launch.json` server config (already set) or an ad-hoc `npm run dev -w nw-tracker-server` — must set **`BACKGROUND_JOBS_ENABLED=0`**: two in-process schedulers on one SQLite file double-run every due sync (duplicate `Sync` rows in `app_messages`, identical diffs) and race writes ("database is locked"). See the env bullet under Server runtime modes.
- **Crypto EOD** (`crypto_eod`): due from 23:55 Chile; `equity_daily` trade dates are **UTC** days. Source: CoinGecko daily USD (includes weekends; Yahoo skipped gaps). Due close = last **completed** UTC day at sync time (never the in-progress UTC calendar day). Stale carryover persists after midnight Chile until that day is in DB (see `cryptoEodDueUtcYmd` in `equityEodSync.ts`). Backfill: `npm run backfill:crypto-coingecko-eod -w nw-tracker-server`.
- Manual “Marcar desactualizado” calls `notifyGlobalSyncScheduler()` to start polling immediately.

## Live market quotes (intraday)

- **Scheduler** (`server/src/liveMarketQuotesScheduler.ts`): polls Yahoo on a fixed interval (default **5 min**). HTTP handlers read **`live_market_quotes` only** — no on-demand Yahoo from client requests. The tick also runs `ensureWatchlistEquityHistoryDepth()` (watchlist.ts): `equity_daily` history to the watchlist YTD/YoY anchors (~400d) for watchlist + composite tickers, one Yahoo attempt per ticker per Chile day — `GET /api/watchlist` serves DB only and never fetches.
- **Manual backfill:** `npm run live-quotes:sync -w nw-tracker-server`
- **Env:** `LIVE_QUOTES_SYNC_ENABLED` (default on), `LIVE_QUOTES_INTERVAL_MS` (default `300000`), `LIVE_QUOTES_RETENTION_HOURS` (default `48`), `LIVE_QUOTES_MAX_AGE_MS` (default `2 × INTERVAL_MS`).
- **EOD** (`stocks_nyse` / `crypto_eod` in global sync) still writes `equity_daily` after the close; live quotes are for intraday MTM / marquee / current-month P/L.
- **Cache invalidation:** same-connection sync writes don't bump `data_version`, so both the live-quotes poll (when a value actually changed) and `runGlobalSyncAll` (when a live run applied changes) call `invalidateMarketDataAggregations()` (`aggregationCache.ts`) — drops `account.monthly_perf|` / `group.consolidated_monthly|` / `group.valuation_closing_by_date|` / `dashboard.page_bundle|` / `daily.series|` (keeps `cc.billing_detail|`) and notifies the warmer. Otherwise dashboard bucket totals / overview live point stay pinned at cache-build-time prices while per-account rows track live marks (they read quotes fresh per request).
- **USD/CLP:** During NYSE regular session, scheduler fetches Yahoo **`CLP=X`** into `live_market_quotes`; MTM/marquee use that via `fxForLiveMtm`. After **17:30 Chile**, all CLP↔USD conversions read Yahoo CLP=X EOD in **`fx_daily`** (NYSE trade dates; synced by `yahoo_fx_usd`). Banco Central dólar observado is stored separately in **`fx_daily_bcentral`** (reference only; synced by `sbif_usd` from 18:00 Chile). Backfill: `npm run backfill:yahoo-fx-usd -w nw-tracker-server` (conversions) and `npm run backfill:sbif-fx-eur -w nw-tracker-server` (BCentral reference + EUR).

## Daily series — calendar-day grid deltas (`dailySeries.ts`)

- **`getBucketDailySeries(accounts, { unit, days, now?, includeAccounts? })`** (`server/src/dailySeries.ts`): one point per **Chilean calendar day** ending today (`chileWallClockAt(now).ymd`; `days: 0` = since `portfolioStartYmd`, `DAILY_SERIES_MAX_DAYS` 7400 is a sanity bound, not a product cap — full-history daily is a user choice, 2026-07-21). Values are true levels via `bucketValueInUnitAt` (→ `accountMarkClpAtYmd`) — each account's mark is flat on its own closed days (weekend/holiday), so **per-account market-calendar attribution falls out of level deltas** (a `.SN` line moves on Chilean days, USD stocks on NYSE days, crypto/UF every day) with no session-window bookkeeping. Each point carries **`market_day`** (`isNyseTradingDay || isChileBusinessDay`; false = weekend/shared holiday — the detalle table dims those rows). Flows use the same deposit-event accounting as `netDepositFlowBetween`, bucketed per `(prev_day, day]` (regular accounts: bulk `loadMergedDisplayDepositInflowEvents`; USD-cash: telescoped `balance − interest`). `pl = delta − flow`; missing legs → null, never a fake 0. Grid helper: `chileCalendarDaysListEndingAt` (in-file; the old `nyseSessionsListEndingAt` was removed).
- **Cached wrapper** `getBucketDailySeriesCached(scopeKey, …)` (real clock only) → `daily.series|<scope>|<unit>|<days>|<ids>` keys, dropped by `invalidateCcBillingDetail` (the account/CC write funnel) and `invalidateMarketDataAggregations` via `invalidateDailySeries()`.
- **Book-value carry for manual-marked accounts** (`storedMarkFlowCarry.ts`): the terminal stored-`valuations` forward-fill (historical branch in `accountMarkClpAtYmd` + both generic stored returns in `syncLatestDisplayValueClp`) returns stale mark **+ net personal flows since its date** (`netDepositFlowBetween`, the same reference as the daily/d1 flow legs) — so a deposit day steps the value with `pl = 0` by construction, and the next mark day carries the clean inter-mark P/L. A mark dated the flow day has an empty carry window (no double count). **Never applied to CC or mortgage balances** (kind-gated: CC owed-on-date has its own derivation; a mortgage payment's balance effect is amortization only). This also makes the Rentabilidad d1/w1 cells and today's dashboard values deposit-aware for these accounts. Same pass fixed `syncLatestDisplayValueClp` ordering: depto property/mortgage UF marks now resolve **before** the stored return (as in the `accountMarkClpAtYmd` today branch) — a stale property valuation no longer shadows the live ledger mark in allocation pies/live points.
- **Known limits:** monthly-marked accounts (DAP, cuenta ahorro) still render P/L as steps — the mark day carries the accumulated inter-mark P/L (carry only moves flows, never fabricates intraperiod returns). Cold-build costs (real DB, 2026-07-21): 90d ≈ 1.3s, 1y ≈ 2.1s, full history (~3.7k days × 31 accounts) ≈ 19s — cached afterwards, but live-quote ticks re-invalidate, so a Total-range daily view re-pays the build each poll (idea if it hurts: only re-price the live tail). The depto property/mortgage ledger is memoized (`depto.ledger|rows`, dropped by `invalidateCcBillingDetail`); remaining hot spot is CC `latestCreditCardValuationRowAsOf` (~400ms on ·1617).
- **Day period view (2026-07-20, phase 2):** `metricsPeriod` is `"day" | "month" | "year"` (toolbar Período: Diario/Mensual/Anual). Dashboard rows carry `deposits_day_* / delta_day_* / prior_day_close_*`, anchored at **plain yesterday (Chile calendar) for every account** (2026-07-21; the per-class `dayWindowAnchor.ts` machinery was deleted — marks are flat on each account's closed days, so per-account market calendars fall out of the level delta with no anchor table). Mortgage keeps its loss-negative convention `prior − close − payments`; CC `prior_day_close` is real but `delta_day` stays null by convention — a CC balance move is flows, not P/L, and daily financing cost is not modeled. Crypto rows mark **live** — `getAccountPositionMeta`'s crypto branch is live-first like `equityBrokeragePositionMeta` (a raw-EOD position value pinned rows at yesterday's close and zeroed the day delta, since the calendar-day anchor mark IS that close). Bucket totals carry `prior_closes.day` + `day_end` (per-account anchors via `dashboardBucketDayPriorCloses`), and `dashboardNavCardMetrics` emits a `day` variant per node (inversiones hub composes day from child buckets — the consolidated slice is monthly). **Month-grain trap:** `buildDashboardBucketValueTotals` maps the consolidated *monthly* closing onto every date of its month — never use it for daily grids; per-session values come from `buildDashboardBucketDailySeriesClp` (Σ raw marks + cash_eqs CC netting), which also feeds `GET /api/dashboard/overview-daily` (`daily.overview|` cache keys) and the day prior-closes; the daily payload carries **`invested`** (retirement + brokerage level) and **`liabilities`** (Σ liability-account marks — mortgage balance + CC owed-on-date) per point (2026-07-22), with the monthly builder's leading-null convention for real_estate/liabilities, so the day-mode overview chart shows the same line set as monthly. Client: pickers three-way (`periodMetricsFromVariant`), month/year-only surfaces clamp via `monthYearMetricsPeriod` (flow pages), the overview chart swaps to the daily series (`primaryXAxisGranularity="day"`: densify/zero-anchor pass-through, month-boundary axis ticks, ISO tooltip titles), nav-snapshot cache bumped to v7.
- **CC per-day owed + daily extras (2026-07-21, phase 4):** between stored anchors, `accountMarkClpAtYmd`'s CC branch returns anchor + **post-anchor signed line activity** (charges +, PAGOs −, `postCloseLiveBalanceAdjustmentClp` — the same evidence stream the month-end writer applies post-cierre; installments enter at anchors, cupo-based). The normalized line stream is memoized under the `cc.billing_detail|<id>|postclose_lines` satellite key (per-account `invalidateCcBillingDetail` prefix-deletes `<key>|`). **Header-only payments (2026-07-22):** the current Santander CLP format prints the previous facturación's payment as a dated «MONTO CANCELADO» row that the parser drops by design (the amount lives in the statement header) — so the stream synthesizes the PAGO event from `cc_statements.monto_pagado_anterior` + `monto_pagado_anterior_date` (migration 166; parser captures the printed date when exactly one body row matches the header amount). - **Strip 1D/1W + ticker displayPL (2026-07-21):** the Rentabilidad short-horizon cells are calendar-framed — 1D = today (live) vs **yesterday**, 1W = vs 7 calendar days ago (`periodReturnsShortHorizon.ts`; the NYSE display-session framing and `d1_is_live` are gone — MTD keeps its live badge). - **Group/account daily views (phase 3):** `GET /api/daily-series?portfolio_group=|account_id=&unit=&days=` serves the bucket daily series **plus per-account chart lines** (`getBucketDailySeries({ includeAccounts })` — same marks, zero extra cost; group scope resolves via `listAccountsForGroupTab`, so Pasivos/CC issuer tabs work too). ## Quote currency / Santiago (`.SN`) tickers

- **`equity_daily` stores `close` + `currency` ('usd' | 'clp')** (migration 152); `live_market_quotes` rows are `kind='equity'` + `currency`. Writers stamp `equityQuoteCurrency(ticker)` (`server/src/equityQuote.ts`, single source of truth: `.SN` → clp, else usd); readers **throw on stored-currency mismatch** — fix rows, don't coerce.
- **`.SN` = Bolsa de Santiago = `EquityMarketKind "santiago"`**: session = Chile calendar day (live window ≈ 09:30–17:05 Chile weekdays); EOD display = latest bar ≤ Chile today (on-or-before absorbs Chilean holidays).
- **MTM**: clp-quoted → `value_clp = units × close`, **no fx** (`computeEquityMtmClp`, `equityBrokeragePositionMeta`, `ccInvestmentProxy`, `marketSeries` derives `equity_usd = close / clp_per_usd`). USD composites throw if a clp ticker sneaks in (`watchlistComposite`).

- **CLP-funded trades**: `stock_buy`/`stock_sell` on a clp-quoted stock transfer against **CLP cash** with `amount_clp` (no `amount_usd`; enforced in `validateBrokerageTransferEndpoints`). They count as `clp_wire` capital flows (`equityBrokerageCapitalFlows`) and the funding CLP-cash leg feeds the aportes line (`accountDeposits`). `dividend_payout` stays USD-only (throws for clp stocks until needed).
- Client: account summary / detail-bundle expose **`equity_quote_currency`**; the stock movement form shows Monto CLP + CLP-cash counterpart for clp-quoted tickers (`stockQuoteCurrencyForTicker`, `brokerageFlowKindNeeds{Clp,Usd}ForQuote`).

Older history: **`npm run backfill:bolsa-santiago-eod -w nw-tracker-server`** pulls daily closes per `.SN` ticker from the Bolsa de Santiago point-history API — up to ~10 years, capped at the instrument's listing date (`server/src/equityBolsaSantiagoEod.ts`; run 2026-07-03, filled CFIETFIPSA.SN 2025-05-12 → 2026-07-02 = full history since its Mayo 2025 inception).

## Deferred: value + currency refactor

**Goal (decided 2026-07):** reduce per-currency *code branching* with straightforward `amount` + `currency` splits where the new shape propagates through readers — not storage-pattern consistency for its own sake. A DTO/derived field named `value_clp` that is definitionally a CLP display value is **not** a violation (most of the ~360 `value_clp` string hits are this); the convention targets DB storage plus parallel `_clp`/`_usd` logic branches. Storage precedent: `equity_daily` / `live_market_quotes` (migration 152) — readers select `value` + `currency` and **throw on unexpected currency**, never coerce. Migrate opportunistically when touching an area — no big-bang renames.

Tranche status, by measured surface:

- **`valuations` — phase 1 done (migration 154):** `value` + `currency` (always 'clp' today); every reader selects both and throws on non-clp via `assertValuationCurrencyClp` (`server/src/valuationValue.ts`) — grep those calls to find the sites phase 2 must widen. DTO field names stay `value_clp` (definitionally CLP after the guard).
- **`movements.amount_clp/amount_usd` — the monster (847 + 223 non-test hits, 86 files;** heaviest: `flowsCheckingGastos`, `movementUnitsPolicy`, deposits reconciliation). Strictly opportunistic; many rows legitimately carry both legs (fx conversions), so this needs its own design pass, not a mechanical split.
- **`cc_billing_month_balances` + `credit_card_account_config.cupo_*` — folded into the future multi-currency-card (EUR) project;** do not do standalone. The EUR project's critical path is elsewhere: `statementSlotsByBillingMonth`'s literal `{clp, usd}` slot model → per-currency map, `cc_statement_lines.amount_clp/amount_usd` (note `amount_orig` + `orig_currency` already exist), and EUR conversion helpers over `eur_daily` (`clp_per_eur`, synced by `sbif_eur` — today a reference series read only by rates display). ## Real-estate expenses (`/flows/expenses/real_estate`)

- **Model:** `expense_accounts` = tracked places, **fully data-driven** (no slug unions or hardcoded lists in code): occupancy in `active_from`/`active_to` (YYYY-MM), `property_account_id` FK → the net-worth property master (owned places), `comunidad_merchant_patterns` (comma-separated place-specific merchant patterns). Values are personal data, set in the live DB only. New rental = «Añadir lugar…» on the page (POST `…/places`); new property = same form with the property account picked — mortgage rows then attach automatically. Payload `places[]` drives sections, chart series (dynamic slug keys + indexed palette), and labels. Bills = `expense_entries` rows with `expense_account_id` set: kind in `category` (`RealEstateBillKind`, incl. `rent`/`water`), consumption in the **`kwh` / `m3` columns** (migration 161 promoted the legacy `kwh=`/`m3=` note tags; notes are human provenance). `real_estate_expense_links` ties a bill 1:1 to the CC/checking purchase that paid it (auto by merchant patterns + amount within bill+0/+2 months, or manual).
- **Purchase-first assign** (`POST /api/flows/expenses/real-estate/assign`, «Asignar compra…» per place): creates the bill FROM an unlinked purchase (amount/month from the purchase) plus the manual link in one tx — how uncovered history (el vergel, rents) gets linked without inventing amounts. The assign modal pool (`GET …/unlinked-purchases`) defaults to category `bills` («Cuentas y servicios») scoped to the place's occupancy period (+2-month tail), kind-pattern matches float up, multi-select bulk-assigns runs of months. Consumption editable per row (`PATCH …/entries/:id/consumption`); billed month editable per row (`PATCH …/entries/:id/bill-month` — rewrites the month-end `spent_on`, rejects months that would put the linked purchase outside the +0..+2 window) and the assign modal has a «mes anterior» toggle (bill_month = purchase month − 1); `DELETE …/entries/:id` removes a bill outright (link/rejections cascade). Bulk history: `npx tsx scripts/suggest-real-estate-links.ts` (report-first, `--apply`; contribuciones + same-day/amount dups + odd-amount rents are review-only; never MUTUARIA/METLIFE/TOKU mortgage cash; cuota-split lines dedupe by purchase_key). Ran 2026-07-10 (~220 links: el vergel 2019–2021 full incl. synthetic gap rows, lastarria, suecia).
- **Mortgage rows are read-only ledger derivations** (suecia): kind `mortgage`, `expense_entry_id: null`, amount = amortización + interés + seguros from `depto_payments` regular cuotas (pie/prepagos are capital events, and the movement's `amount_clp` is an equity mark, not the cash paid). Never stored as expense entries. `bill_month` = the cuota's **scheduled month**, derived from the cuota numbers (`mortgageCuotaBillMonths`: modal anchor of month(occurred_on) − cuotaN; throws on inconsistent numbering) — a skipped cuota paid the next month still lands in its own month; `spent_on` keeps the real payment date (shown as a «pagado» hint).

- **No double counting:** the manual-gastos loader (`flowsManualExpenses.ts`) excludes place rows **structurally** (`expense_account_id IS NULL`) — a linked bill's money already flows through its CC/checking purchase line.

## Historical mirror pairs + movement search

- **Mirror pairs** (`/panel/mirror-pairs`): two pre-transfer-model single-leg rows (retiro + same-amount depósito on different accounts, inflow 0–5 days after outflow) are converted into one transfer row dated the **outflow day**; original legs (ids, dates, amounts, units, notes) live in the **`movement_mirror_merges`** table keyed by the transfer movement (ON DELETE CASCADE; the transfer note is a human summary) and conversion is undoable (`undoMirrorConversion`). Candidates: `listMirrorPairCandidates` (`server/src/movementMirrorPairs.ts`) — high confidence = unique both directions + within the cartola business-day window (`bankDateMatchesTransferDate`) + no month straddle; checking-inflow month-straddle pairs are hard-blocked (cartola anchors). Excluded legs: flow_kind/USD legs, both-legs-cuotas, dap either side, afp/afc **inflows** (pre-tax payroll), deposits already in `expense_deposit_links`/`checking_gap_deposit_mirrors`/`payroll_work_earnings`, Buda/ahorro-split notes. Rejections persist in `movement_mirror_pair_rejections` (migration 155; ids cascade on full rebuilds). **Link-established pairs**: 1:1 `expense_deposit_links` rows (auto/manual, `checking-cartola:` keys resolved via `movementForCheckingPurchaseKey`, amounts equal) surface as `linked: true` high-confidence candidates — the link is the evidence, no window applies; the link row cascades away at conversion. Multi-allocation/partial links, synthetic links, and Buda-note deposits stay out. One transfer carries one `units_delta` (abs; cuota readers use `transferLegUnitsThroughDate`). **Month-precision legs** (cuenta_ahorro_vivienda — sheet records mm-yyyy only; movements dated conventional month-end): the pairing window is the whole month ± 7 days across the boundary (`monthPrecisionPairAllowed`) instead of the day-gap rule, the bank-window requirement is waived, and the converted transfer takes the **real-day (checking) leg's date** — so cartola re-import dedupe matches same-day and checking anchors are untouched (the checking-straddle hard-block is skipped when the out leg is month-precision). Conversion is balance- and aportes-neutral: the aportes/monthly-P&L reader buckets each side of a merged transfer by its **original leg date** from `movement_mirror_merges` (`loadTransferLegSignedFlowEvents` in `accountDeposits.ts`), so a cross-month month-precision conversion doesn't create a −X/+X monthly-P&L couplet on the month-precision account (its valuation marks carry the flow in the sheet month, not the transfer month); converted cartola credits leave the income pool (a correction) and mirror-merge transfers into checking self-resolve in the deposits reconciliation.
- **Checking↔CC payment mirrors (2026-07-22):** a checking «Traspaso a T. Crédito»/PAGO TARJETA debit pairs with the card's payment evidence — a PAGO/MONTO CANCELADO statement line (legacy formats) or the statement's `monto_pagado_anterior` header (current format) — by exact amount within ±4 days (`ccPaymentMirrors.ts`; candidates in the /panel/mirror-pairs «Pagos de tarjeta» section, POST `/api/movement-mirrors/convert-cc-payments`). Conversion writes one transfer (checking → CC master) dated the **card's credit date** (the immovable-evidence side — the inverse of the outflow-day rule, same spirit as the month-precision exception) with `flow_kind = 'pago_tarjeta'`; the merge row keeps the cartola leg (`out_*`) and references the evidence (`in_statement_line_id` / `in_statement_id` — migration 167 made `in_movement_id` nullable with a CHECK that exactly one in-ref is set). Statement evidence is never touched (the CC owed walk keeps reading it; the transfer's card leg is inert — CC valuations ignore movements), and `pago_tarjeta` transfers emit **no deposit/aportes events** (paying your own card is internal to the CC-netted cash bucket; `accountDeposits` skips the kind, so the daily/monthly P/L is zero by construction). Undo restores only the checking leg. All 63 date-skewed historical pairs (bank credits the card 1–4 days before the cartola debit — each a one-to-three-day cash bump in the daily view) were converted 2026-07-22; same-day pairs remain optional candidates.
- **Movement search lives in the flows tables** (the standalone `/search` page was removed; `/search` redirects home). `FlowsTable`'s filter bar carries the full set — year/type/account/category, `q` (note + account + counterpart + flow-type label), `date_from`/`date_to` (inclusive), `amount_exact` XOR `amount_min`/`amount_max` (rounded |amount_clp|; exact disables min/max) — on every consumer: the **net-worth dashboard is the master view** (`/api/groups/net_worth/flows`), plus group and account pages. Shared filter logic: `applyFlowFilters` + `parseExtraFlowsFilterParams` (`server/src/flowsApi.ts`), parsed by both flows routes in `routes/dashboard.ts`. Scope is **movements only** — CC purchase lines live in `cc_statement_lines` and are searched from the Expenses page.

## Projections (/projections)

- `GET /api/projections` (`server/src/projections.ts`, pure engine `runProjectionEngine`): one **real** (today's-money) trajectory in the display unit — accumulation to age 65 (born 1992-01 → retiro 2057-01; invested compounds monthly + aporte, RE/cash hold real value flat), then three drawdown strategies over the **drawdown pot** = invested + `liquidate_other_pct`% of the non-invested remainder (default 0 — FIRE framing; 100 ≙ total NW, anything between = partial RE sale at 65) to `end_age`. `monthly_rent_clp` (today's money) adds passive income to every strategy; the fixed-income strategy targets TOTAL income so its pot only funds `target − rent` (SWR fixed-real %, % of current balance, fixed real monthly income; depletion ages in `summary`). Nominal line derives from the real path with the unit's inflation — both currency views share the trajectory (independent nominal paths would drift). ## Import sources vs runtime

**Rule:** files under `cfraser/` are **import inputs only**. HTTP, dashboard, valuation, and sync read **SQLite**; missing data → error or “run import”, not a silent file fallback.

- **Depto:** runtime reads the **movement ledger** — `loadDeptoLedgerFromMovements()` joins property-account movements with **`depto_payments`** (one row per movement: cuota + the ~17 payment fields; UF-day/deltas/acumulados/analytics re-derived from `uf_daily`). ## Server runtime modes (local vs hosted demo)

One binary, env-driven (`server/src/httpSecurity.ts`, root `.env` loaded at boot):

- **`HOST`** — bind host; default `127.0.0.1` (keeps the unauthenticated local API off the LAN). Hosted demo (Render) sets `HOST=0.0.0.0`.
- **`CORS_ALLOWED_ORIGINS`** — comma-separated allowlist; defaults to Vite dev/preview origins (`localhost:5173`/`4173`). Never use a reflect-any-origin CORS config — any website could read the API.
- **`AUTH_PASSWORD`** — when set, every `/api` route (except `/api/health` and the `/api/auth/*` login endpoints) requires a valid **signed session cookie** (`nw_session`), issued by the in-app `/login` page: any **syntactically valid email** (format check only) + this shared password (recruiter-demo model). No HTTP Basic / browser prompt — anonymous `/api` requests get a plain `401` (no `WWW-Authenticate`), and the client's route guard redirects to `/login?next=…`. The cookie is a stateless HMAC token (`server/src/httpSecurity.ts` `issueSessionToken`/`verifySessionToken`; secret = `SESSION_SECRET` or derived from `AUTH_PASSWORD`; 7-day max-age → re-login), so it survives the demo's per-deploy DB regeneration. Non-`/api` paths (the SPA shell + `/assets`) are never gated, so the login page can load. Auth endpoints live in `server/src/routes/auth.ts`; in **`DEMO_MODE`** the shared password is returned by `/api/auth/status` and shown/prefilled on the login page (recruiter self-serve — the password is intentionally public; gated on demo mode so a non-demo `AUTH_PASSWORD` deployment never leaks it). Authenticated emails are logged to **`demo_auth_logins`** (one row per email+day, with first/last seen and request_count), written at login + per authenticated request. Unset = local mode, no auth, no logging.
- **`DB_BACKUP_ENABLED` / `DB_BACKUP_KEEP` / `DB_BACKUP_DIR` / `DB_BACKUP_DIR_KEEP`** — daily auto-snapshot of the DB while the server runs (`dbBackupScheduler.ts`; default on, keep 14 local `auto-daily` snapshots, optional second copy dir keeping 60). `DB_BACKUP_DIR` points at `~/Documents/backups/nw-tracker` (set in root `.env`). Manual `npm run db:snapshot` labels are never pruned.
- **`CACHE_WARM_ENABLED`** — proactive dashboard cache warmer (`dashboardCacheWarmer.ts`; default on, `0` disables). Rebuilds the page-bundle caches in the background at boot, at 00:05 Chile (day-rollover clear), when `data_version` shows an external write, and debounced (30s) after in-process invalidations — so no interactive request pays a cold aggregation build. The warmed object itself is the served response (`dashboard.page_bundle|<unit>` in the aggregation cache), not just its inner aggregations. CC billing detail lives in the aggregation cache (`cc.billing_detail|<accountId>`; day/data_version fresh + invalidated by CC write hooks), not rebuilt per request.
- **`BACKGROUND_JOBS_ENABLED`** — umbrella for the four in-process schedulers (`server/src/backgroundJobsEnv.ts`, applied at boot after demo-mode defaults): `0` defaults `GLOBAL_SYNC_ENABLED` / `LIVE_QUOTES_SYNC_ENABLED` / `DB_BACKUP_ENABLED` / `CACHE_WARM_ENABLED` to `0`; an explicitly set per-job flag wins. Set on every `.claude/launch.json` server config so agent-spawned dev servers never run syncs/backups against the shared DB — only the user's primary app instance does (see Global sync scheduler).
- Express async route handlers must be wrapped in `asyncHandler` (defined in `index.ts`) — Express 4 does not forward async rejections, and an unhandled rejection kills the process. A terminal error middleware turns route throws into JSON 500s.

## Schema baseline (fresh DBs)

`initSchema()` executes `server/src/schemaBaseline.ts` — the full schema dumped from the live DB (as of migration 155), all `IF NOT EXISTS`. On a brand-new DB it marks migrations ≤ 155 as pre-applied. **Migrations ≤ 155 were squashed into the baseline 2026-07-07 and their files deleted** (personal-data ones also purged from git history; copies of everything live outside the repo in `~/Documents/backups/nw-tracker/{removed,squashed}-migrations/`). The only pre-baseline files kept are the 8 reference-row seeds in `BASELINE_REFERENCE_DATA_MIGRATIONS` (db.ts) — idempotent category/uniqueness seeds the schema-only baseline skips, which still run on fresh DBs. A recorded-but-missing file is harmless to the runner (the live DB keeps all 155 ids in `schema_migrations`), so numbering gaps are expected; post-baseline migrations (156+: fintual flow_kind, depto_payments + movement_mirror_merges + staging drop, accounts.import_key) apply normally everywhere. A migration may pair with a `POST_MIGRATION_HOOKS` entry in `db.ts` (pure TS module, runs inside the migration transaction — see `legacyNoteBackfills.ts` for 157). Tables dropped by a migration must ALSO be removed from the baseline, or every boot resurrects an empty shell via `IF NOT EXISTS`. When squashing again: `npx tsx scripts/regenerate-schema-baseline.ts --last <newest-migration>` (from `server/`), then delete the newly-covered files (keep the reference seeds). Migration SQL is naively split on `;` (no string-literal lexing) — no triggers or `;`/`--` inside literals; use a post-migration hook in `db.ts` for those.

## Dashboard performance profiling

- **`npm run profile:dashboard`** (from `server/`): sequential timings for dashboard builders on the test/dev DB (`NW_TRACKER_TEST_DB` or default `server/data/nw-tracker.test.db`).
- **`DEBUG_PERF=1`** (or **`DEBUG_VERBOSE=1`**) on the server logs `[heavy] …ms` spans for labeled work (`server/src/heavyWork.ts`) during HTTP requests.
- Home page: **`GET /api/dashboard/page-bundle`** — one client call replaces the old 5-way fan-out. The whole bundle response is served from the aggregation cache (`dashboard.page_bundle|<unit>`, built by the warmer; cached value is the in-flight promise so concurrent cold requests share one build). Every explicit invalidation drops it — `invalidateCcBillingDetail` and `invalidateMarketDataAggregations` are the two funnel points — so a warm request is ~5ms; only requests landing in an invalidation→rewarm gap pay the full build.
- Nav strip (sidebar / group pages): **`GET /api/dashboard/nav-context`** — accounts + liabilities + overview in one call (client fetches only on group/liabilities routes and account detail when the nav node has routable children; home uses `page-bundle` only).
- Dominant cost is usually the portfolio-group totals aggregation inside **`getDashboardValuationTimeseries`** / **`getGroupValuationTimeseries`**; `@heavy` marks those entry points in source.

## Server Vitest (SQLite)

- Vitest uses **`server/data/nw-tracker.test.db`**, never `nw-tracker.db`. Both are gitignored (`server/data/*.db`).
- **`NW_TRACKER_TEST_DB`** is set in `server/vitest.config.ts` (`test.env`) and in the server **`npm run test`** script so it is present before `db.ts` loads.
- **`server/src/vitest.globalSetup.ts`**: when `nw-tracker.test.db` is missing, generates the **lean synthetic preset** (`generate-demo-data.ts --preset=test`) — tests must not depend on personal data. Delete `nw-tracker.test.db` to rebuild; the file is never refreshed automatically, so a stale pre-synthetic dev-DB copy on disk silently keeps being used (symptom: suites failing on missing `demo:*` fixture accounts or real-data invariants — delete the test DB). Escape hatch: `NW_TRACKER_TEST_FROM_DEV=1` copies the local `nw-tracker.db` instead (reproducing an issue against real data).
- **`server/src/vitest.setup.ts`** → **`server/src/test/vitestDbSeed.ts`**: ensures an isolated Santander CC master `credit_card_master|santander|vitest-fixture` (never a real card) for merge/import tests.
- Override the test file with `NW_TRACKER_TEST_DB` (absolute path, basename under `server/data/`, or `:memory:`). Vitest uses `pool: "forks"` with one sequential fork (`fileParallelism: false`) — better-sqlite3 segfaults nondeterministically in worker threads at teardown, killing the run after a few files; parallel forks would race migrations on the single SQLite file.
- To remove Vitest fixture rows from **`nw-tracker.db`** (e.g. after running tests against the wrong DB): `sqlite3 server/data/nw-tracker.db < server/scripts/cleanup-cc-vitest-pollution.sql`.

## UI tables — desktop/mobile sync

Tables in the client have parallel desktop and mobile renderings. The desktop version uses `<td className="desktop-only">` columns; the mobile version uses `<TableMobileCard>` / `<TableMobileCardRow>` inside a `<td className="mobile-only">`. When adding, removing, or reformatting a column on one, apply the identical change to the other. Both variants are usually in the same component or a companion file (e.g. `CreditCardSections.tsx` + `CreditCardPurchaseMobileCard.tsx`).
